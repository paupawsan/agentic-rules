# Agentic Rules Framework向けナレッジグラフ実装ガイド
**フレームワークバージョン**: 1.4.0
**ドキュメント目的**: KG機能向けエージェント実装ガイド
**実装アプローチ**: 論理アルゴリズムと擬似コードのみ

## 🎯 **コア問題ステートメント**

ユーザーがagentic-rulesフレームワークをクローンするとき、彼らは期待します：
- ✅ **会話間のメモリ永続性**
- ✅ **関係理解のためのナレッジグラフ構築**
- ✅ **会話での自動KG使用**（手動活性化ではない）
- ✅ **高度な機能が利用できない場合のフォールバックメカニズム**

**課題**: 物理スクリプトを含めることはできません。エージェントは論理アルゴリズムと擬似コードのガイドを通じてKG機能を実装する必要があります。

## 🏗️ **フレームワーク設計原則**

### **KG統合アーキテクチャ**
```
ユーザー会話 → メモリシステム → KGスキャナー → コンテキストインジェクション → 応答
       ↓               ↓              ↓              ↓              ↓
   自然入力   自動キャプチャ   関連性チェック   自動拡張   拡張出力
```

### **主要設計要件**
1. **ゼロ設定**: KG機能がすぐに動作
2. **自動発見**: 手動の「KGを使用」コマンド不要
3. **フォールバック弾力性**: 基本メモリで動作、KGで強化
4. **段階的改善**: 基本 → 中間 → 高度KG機能

## 🤖 **エージェント実装アルゴリズム**

### **アルゴリズム1: KG対応メモリ作成**
**目的**: すべてのメモリが既存のKGを自動的に参照することを保証

```
アルゴリズム: Create_KG_Aware_Memory
入力: user_input, conversation_context, memory_category
出力: kg_enhanced_memory_entry

ステップ:
1. 標準フィールドで基本メモリエントリを作成
2. 既存のメモリシステムをKGデータ用にスキャン:
   - プロジェクトメモリで"kg_data"、"kg_analysis"、"kg_relationships"を検索
   - フレームワークKG情報用に共通メモリを確認
   - メモリディレクトリ構造でKG関連ファイルを検索

3. KGデータが見つかった場合:
   - 関連するエンティティと関係を抽出
   - 関連性スコアを以下に基づいて計算:
     * 現在の会話との意味的類似性
     * KGデータの時間的最近性
     * 現在のコンテキストとのトピック重複
   - 関連性しきい値以上のKG項目をフィルタリング（デフォルト: 0.6）

4. メモリエントリを強化:
   - 上位N個の関連KG項目で"kg_references"フィールドを追加
   - 関係インサイトで"kg_context"フィールドを含める
   - 接続された概念用に"kg_relationships"フィールドを追加

5. KG相互参照付きの強化メモリを保存
6. 成功確認を返す

フォールバック: KGデータが見つからない場合、標準メモリエントリを保存
```

### **アルゴリズム2: 会話KGコンテキストインジェクション**
**目的**: 新しい会話に自動的に関連するKGインサイトを含める

```
アルゴリズム: Inject_Conversation_KG_Context
入力: new_conversation_request, user_context, project_context
出力: kg_enhanced_conversation_context

ステップ:
1. 会話リクエストを分析:
   - 主要エンティティ、トピック、意図を抽出
   - プロジェクトコンテキストとユーザーの目標を特定
   - リクエストに関連する知識ドメインを決定

2. メモリシステムをKGデータ用にクエリ:
   - トピック関連KG用のプロジェクトメモリを検索
   - 一般的なフレームワークKG用の共通メモリを確認
   - ユーザー固有のKGパターンを検索

3. KG関連性を計算:
   見つかった各KG項目に対して:
     relevance_score = calculate_relevance(
       conversation_entities,
       kg_entities,
       temporal_recency,
       topic_overlap
     )
   ENDFOR

4. 上位KGコンテキスト項目を選択:
   - 関連性スコアで降順ソート
   - max_context_itemsまで制限（デフォルト: 5）
   - 高インパクト関係を優先

5. インジェクション用にKGコンテキストをフォーマット:
   - KG関係を自然言語に変換
   - 関連接続の文脈的サマリーを作成
   - 関係についてのインサイトステートメントを生成

6. 会話にKGコンテキストをインジェクト:
   - システムコンテキストに関連KGインサイトを追加
   - 利用可能なコンテキストに関係サマリーを含める
   - 応答生成用にKGデータにアクセス可能にする

フォールバック: KGデータが利用できない場合、標準会話コンテキストで続行
```

### **アルゴリズム3: 段階的KG構築**
**目的**: 高度なスクリプトを必要とせずにKGを段階的に構築

```
アルゴリズム: Progressive_KG_Construction
入力: conversation_data, memory_entries, project_context
出力: incrementally_built_kg

ステップ:
1. 基本エンティティ抽出から開始:
   - 名詞、固有名詞、技術用語を識別
   - 会話から主要概念を抽出
   - シンプルなエンティティリストを構築

2. 基本関係を確立:
   - 一緒に言及されたエンティティを接続
   - 時間的関係を記録（前/後）
   - インタラクションパターンを記録

3. メモリ統合で強化:
   - 既存のメモリエントリと相互参照
   - 繰り返しエンティティ組み合わせを識別
   - 頻度を通じて関係を強化

4. KGを段階的に保存:
   - 構造化されたマークダウンファイルとして保存
   - 一貫した命名を使用: YYYYMMDD_entity_relationships.md
   - 将来の検索用にメタデータを含める

5. 段階的改善:
   - エンティティ共起から開始
   - 意味関係を追加
   - 文脈的メタデータを含める
   - 関係強度スコアを構築

フォールバック: KG構築が失敗しても会話データを強化されたメモリとして保存
```

### **アルゴリズム4: KGクエリ改善**
**目的**: KG関係を使用して情報検索を改善

```
アルゴリズム: KG_Enhanced_Query_Processing
入力: user_query, available_context, memory_system
出力: kg_enhanced_response_context

ステップ:
1. 標準クエリを処理:
   - キーワードと意図を抽出
   - メモリシステムで関連情報を検索
   - 従来のコンテキストを収集

2. KG関係で強化:
   KGデータが利用可能な場合:
     - クエリで言及されたエンティティを見つける
     - 接続された概念を見つけるためにKG関係を探索
     - 間接的に関連する情報を取得
     - 関係コンテキストを含める

3. KGインサイトで結果をランク付け:
   - クエリエンティティに接続された結果をブースト
   - 関係説明を含める
   - 文脈的理解を提供

4. KG強化応答を生成:
   - 応答に関係インサイトを含める
   - 概念間の接続を説明
   - より広いコンテキスト理解を提供

フォールバック: KG強化が利用できない場合、標準クエリ結果を返す
```

## ⏳ **時間認識（バイテンポラル）ナレッジモデル** *(v1.5.0)*

知識は変化します。デプロイ先は移り、慣習は置き換えられ、決定は覆されます。事実を*追加*するだけのKGは、やがて古くなった知識をその後継より上位にランク付けするようになります——最優先の回答が誤った回答になり得るのです。テンポラルモデルはこれをデータ層で解決します: **知識は決して削除されず、置き換えられる（supersede）のです。**

### **モデル**

すべてのノードは、互いに独立した2つの時間次元を持ちます:

| フィールド | 次元 | 意味 |
|-----------|------|------|
| `created_at` | トランザクション時間 | レコードがKGに登録された時点 |
| `expired_at` | トランザクション時間 | レコードが*誤った記録*として廃止された時点（`NULL` = 有効） |
| `valid_at` | イベント時間 | 事実が現実世界で真になった時点（デフォルトは `created_at`） |
| `invalid_at` | イベント時間 | 事実が真でなくなった時点——置き換え時に自動設定（`NULL` = 現在も真） |

モデルのルール:

- **編集せず、置き換える。** 知識が変化したら、新しいノードを追加し、古いノードへ `supersedes` エッジを張ります。古いノードは自動的に無効化され、デフォルトの検索結果から外れます——ただし内容はそのまま残ります。
- **デフォルトビュー = 現在。** クエリは*今*有効な知識だけを返します。置き換え済み・廃止済み・まだ有効になっていないノードは除外され、`supersedes` エッジは、隠された内容を再注入する代わりに注釈付きポインタ（`-> supersedes: old-id [invalidated <date>]`）として表示されます。
- **タイムトラベル。** `as_of=<ISO date>` は、その時点で何が真だったかを再構築します——過去の決定が、当時利用できた知識のもとで*なぜ*下されたのかを監査するのに役立ちます。`include_expired=true` はすべてを表示し、`[SUPERSEDED by …]` / `[expired]` のマーカーを付けます。
- **後継のない廃止。** 事実が単に真でなくなった（何も置き換えない）場合は `invalid` として廃止します。レコードが最初から誤っていた場合は `expired` として廃止します（その瞬間以降は履歴ビューからも隠れます）。どちらも取り消し可能です（`restore`）——何も破壊されません。
- **タイブレーカーとしての新しさ。** ランキングには、最終更新時刻に対する*緩やかな*指数減衰（デフォルトの半減期は90日）を加えられます。僅差の順位を入れ替える程度に抑えられており、関連性を上回ることはありません。

### **使う理由——そして使わない場合**

**メリット:**
- **古い知識が勝ち続けなくなる。** このモデルの動機となった障害はこうでした: 優先度の高い古い事実が、誰かがそのテキストと優先度を手作業で書き換えるまで、後継の事実より上位にランクされ続けたのです。置き換え（supersession）があれば、現在の事実が表に出て、古い事実は注釈付きポインタになります——手動の降格も、履歴の書き換えも不要です。
- **監査可能性。** 「6月10日の時点で私たちは何を信じていたか？」という問いに、クエリで答えられます。過去の決定を、その当時の知識に照らして判断できます。
- **安全な矛盾。** 矛盾する2つの事実は、（`contradicts` エッジが対立を記録した上で）一方が置き換えによって勝つまで共存できます——正しいかもしれない知識を早まって削除することはありません。
- **可逆性。** 誤った廃止は `restore` 1回で元に戻せます。削除されたノードは永遠に失われます。

**コスト / 見送るべき場合:**
- **書き込み経路の規律。** エージェントは「編集せず、置き換える」を身につける必要があります——本フレームワークのルールはそれを教えますが、ストアに直接書き込むアドホックなスクリプトはそれを迂回できます。
- **増加。** 履歴は蓄積します（何も削除されないため）。個人/プロジェクト規模のKG（数百〜数万ノード）では無視できる量です。より大きな規模では、古い置き換えチェーンを削除せずアーカイブしてください。
- **追記専用のドメインには不要。** 知識が決して変化しない場合（例: 静的な参照コーパス）、テンポラルフィルタリングは、決して設定しないフィールドを追加するだけです——その場合、デフォルト（`valid_at = created_at`、すべて現在有効）は非テンポラルKGとまったく同じに振る舞うため、コストはわずかですが、得るものもありません。

### **由来とクレジット**

これは**本プロジェクトの発明ではなく、既存手法の適応**です。バイテンポラルモデリング（*有効時間*と*トランザクション時間*の分離）は、テンポラルデータベースにおける長年の実践です（例: SQL:2011 のテンポラルテーブルを参照）。これをAIエージェントのメモリKGに適用すること——置き換えエッジ、非破壊的な無効化、特定時点での検索——を普及させたのは **Zep の Graphiti** エンジンであり、本設計はそこから着想を得ています。本フレームワークが取り入れたのは*概念のみ*です: Graphiti のコード、スキーマ、テキストは一切再利用していません。そのため、サードパーティのライセンスは付属しません。関連するものの方向性が異なる研究として、埋め込みベースのテンポラルKGの*推論/予測*（例: RE-GCN、[arXiv:2104.10353](https://arxiv.org/abs/2104.10353)）があります。こちらはKGのスナップショットから将来の事実を予測するものですが、本フレームワークは意図的にストレージ/検索の層に留まります——キュレーションされた知識を入力とし、決定的な時間認識検索を出力とする層です。

### **データベースを使った実装（markdownの代わりに）**

markdownのKG（上記のティア）は、インフラ不要のデフォルトです。グラフが、grepして読む方式で快適に扱える規模を超えたとき——あるいは複数のプロジェクトやエージェントが同じ知識を必要とするとき——ストアをデータベースへ移します。SQLite で十分であり、開始にサーバープロセスは不要です。

**スキーマ**（SQLite。テンポラルフィールドは、NULL を許容する3つのカラムです）:

```sql
CREATE TABLE nodes (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL CHECK(type IN ('rule','pattern','fact','procedure','gotcha')),
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    scope TEXT NOT NULL DEFAULT 'global',
    tags TEXT DEFAULT '',
    priority INTEGER DEFAULT 5 CHECK(priority BETWEEN 1 AND 10),
    source TEXT DEFAULT '',
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now')),
    valid_at TEXT,      -- event time: fact became true (NULL -> treat as created_at)
    invalid_at TEXT,    -- event time: fact stopped being true (set on supersession)
    expired_at TEXT     -- transaction time: record retired as erroneous
);

CREATE TABLE edges (
    source_id TEXT NOT NULL REFERENCES nodes(id) ON DELETE CASCADE,
    target_id TEXT NOT NULL REFERENCES nodes(id) ON DELETE CASCADE,
    relation TEXT NOT NULL CHECK(relation IN
        ('applies_to','depends_on','part_of','related_to','supersedes','contradicts')),
    weight REAL DEFAULT 0.5,
    PRIMARY KEY (source_id, target_id, relation)
);

CREATE INDEX idx_nodes_invalid ON nodes(invalid_at);
CREATE INDEX idx_nodes_expired ON nodes(expired_at);
-- Optional retrieval upgrades: an FTS5 table over (title, content, tags) for BM25,
-- and a vector table (e.g. sqlite-vec) for semantic search.
```

**可視性述語**（テンポラル動作の心臓部）:

```
current view (default):   expired_at IS NULL AND invalid_at IS NULL
                           AND (valid_at IS NULL OR valid_at <= now)
as-of view (as_of = T):   (expired_at IS NULL OR expired_at > T)
                           AND (valid_at IS NULL OR valid_at <= T)
                           AND (invalid_at IS NULL OR invalid_at > T)
```

この述語は、アプリケーションコード内で候補に対する後段フィルタとして適用してください。アドホックな SQL の文字列比較として書いてはいけません——タイムスタンプ形式の混在（`datetime('now')` とオフセット付き ISO-8601）は辞書順の比較を壊します。すべての書き込みを1つのタイムスタンプヘルパーに通して正規化してください。

**書き込み経路のセマンティクス:**
- `add(..., supersedes=old_id)`——1つのトランザクション内で、新しいノードを挿入し、`supersedes` エッジを挿入し、`old.invalid_at = new.valid_at` を*それが NULL の場合に限り*設定します（既存の履歴は決して上書きしません）。
- 単独で作成された `supersedes` エッジも、同じ方法でターゲットを無効化します。`contradicts` エッジは何も無効化しません（対立を記録し、解決は後で行います）。
- `retire(id, mode)`——`invalid_at`（事実が終わった）または `expired_at`（レコードが誤っていた）を設定します。`restore` は両方をクリアします。

**既存の非テンポラルデータベースをその場で移行するには:** 純粋な `ALTER TABLE nodes ADD COLUMN …` だけを使ってください（テーブルの削除・再作成は厳禁です——再構築すると SQLite の rowid が振り直され、rowid で結合されている FTS/ベクトルテーブルとの同期が静かに崩れます）。その後、一度だけバックフィルします: `UPDATE nodes SET valid_at = created_at WHERE valid_at IS NULL`。移行は冪等（何度実行しても結果が同じ）にし（まず `PRAGMA table_info` を確認）、初回実行の前にデータベースファイルをバックアップしてください。

### **KGをMCPサーバーへアップグレードする——推奨パス**

自然な成熟の道筋は、**markdownファイル → SQLiteデータベース → そのデータベースの前面に置くMCPサーバー**です。次のいずれかに当てはまるなら、MCPへの変換は本当に良い選択肢になります: (a) 複数のプロジェクト、マシン、またはエージェントが同じ知識を必要とする。(b) grep を超える検索品質（BM25/セマンティック/リランキング）が欲しい。見合う理由は次のとおりです:

- **1つの脳、多数のセッション。** すべてのプロジェクトとすべてのエージェントセッションが、同じツールセット（`kg_context`、`kg_query`、`kg_add`、`kg_link`、`kg_retire`、`kg_get_node`、`kg_list`）を通じて同じストアと対話します——プロジェクトごとのコピーが少しずつ食い違っていくことがありません。
- **ルールはすでにMCPに対応。** 本フレームワークのルールファイルは、接続時には `kg` MCP ツールを優先し、未接続時には markdown ストアへフォールバックするようエージェントに指示しています——変換に必要な**ルール変更はゼロ**で、エンドポイントの設定（Claude Code プラグインの `kg_mcp_url`）だけで済みます。
- **検索の強化はサーバー側に閉じる。** BM25+ベクトルのハイブリッド検索、リランキング、テンポラルフィルタリング、新しさによる減衰——これらすべてが、エージェント向けのテキストに一切触れることなく、ツールインターフェースの背後で改善されます。
- **スキーマ検証付きの書き込み。** ツールのパラメータ（型、有効なリレーション、優先度の範囲）は、ファイル編集で守られることを祈るのではなく、境界で強制されます。

変換*しない*方がよい場合: KGが小さい単一プロジェクトで、プロセスを常駐させる気がない場合です——markdown ストアはリポジトリと一緒にバージョン管理され、コードレビューで差分を確認でき、何のインストールも要りません。共有や検索品質が実際に問題になるまで、そのシンプルさを保つ価値があります。
